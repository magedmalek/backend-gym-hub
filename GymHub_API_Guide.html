GymHub User Guide

Business and technical onboarding guide for new developers: what the application does, how the flow works, how to run it, how to register, how to test it, and how every API is used.

1. Product Overview

GymHub is a backend for managing gyms as service providers. A gym owner can create a gym, define services and subscription packages, add employees with permissions, register customers, sell subscriptions, record payments, activate subscriptions, track attendance, consume guest invitations, and sell extra paid services.

The current repository is a Spring Boot REST API. It does not include the frontend UI, but it provides Swagger/OpenAPI and JSON endpoints for a web or mobile client.

3. Roles and Permissions

User Roles

RoleMeaning
SERVICE_PROVIDERGym owner or service provider. Creating a gym makes the current user the owner.
EMPLOYEEStaff account. Must be linked to a gym as an active employee before dashboard actions are allowed.
CUSTOMERMember account. Customers are normally blocked from dashboard operations.

Employee Permission Model

The backend never trusts an employeeId from the frontend as the actor. The actor is always resolved from the JWT user. The gym owner bypasses permission checks. An employee must be active and must hold the needed permission. ADMIN satisfies every employee permission.

Permissions: SELL_SUBSCRIPTION, ACTIVATE_SUBSCRIPTION, REGISTER_ATTENDANCE, REGISTER_INVITATION, SELL_EXTRA_SERVICE, MANAGE_SERVICES, MANAGE_PACKAGES, MANAGE_CUSTOMERS, MANAGE_EMPLOYEES, FREEZE_SUBSCRIPTION, MANUAL_ADJUSTMENT, CLOSE_CASH_DAY, ADMIN.

4. Main Application Flow

  1. Register a user with role SERVICE_PROVIDER.
  2. Login and copy the returned accessToken.
  3. Create a gym using POST /api/v1/gyms. The caller becomes the owner.
  4. Optionally update gym settings: partial payment, activation policy, and entrance methods.
  5. Create employees and assign permissions.
  6. Create services, then create packages that include selected services.
  7. Create customers or link existing platform users as customers.
  8. Sell a subscription to a customer and record initial payment.
  9. If fully paid and policy is IMMEDIATE, the subscription activates automatically. If policy is MANUAL, activate it with the activation endpoint.
  10. Use member code or customer ID to record attendance.
  11. Consume invitations from subscription quota for guests.
  12. Sell extra services independently or record usage of services included in a subscription.

6. How to Run

Prerequisites

  • Docker and Docker Compose.
  • Java 17 and Maven only if running outside Docker.

Environment

copy .env.example .env

For local Docker, keep these important values aligned:

DATASOURCE_URL=jdbc:mysql://mysql:3306/gymhub_db?createDatabaseIfNotExist=true&useSSL=false&serverTimezone=UTC&allowPublicKeyRetrieval=true
DATASOURCE_USERNAME=gymhub
DATASOURCE_PASSWORD=gymhub
SERVER_PORT=8085
CORS_ALLOWED_ORIGINS=http://localhost:3000,http://localhost:8080
JWT note: The README says the JWT secret should be a long Base64 value for production. Use a proper secret before deploying.

Docker Run

docker compose up --build

Stop

docker compose down

Reset Database

docker compose down -v

Verify

docker ps
docker compose logs app -f
curl http://localhost:8085/v3/api-docs

Swagger UI: http://localhost:8085/swagger-ui.html

7. How to Test

Run the automated tests from the project root:

./mvnw test

On Windows PowerShell, if no Maven wrapper exists in your checkout, use:

mvn test

The current focused test file is GymAccessServiceTest. It verifies that owners, active employees, inactive employees, foreign employees, customers, and ADMIN employees follow the expected access rules.

Manual API Test Strategy

  1. Register owner, login, and save the token.
  2. Create gym and save gymId.
  3. Create a service and save serviceId.
  4. Create a package and save packageId.
  5. Create a customer and save customerId and memberCode.
  6. Sell subscription and save subscriptionId.
  7. Record attendance using memberCode.
  8. Test list/search endpoints with ?page=0&size=20.

1. نظرة عامة

GymHub هو Backend لإدارة الجيمات. صاحب الجيم يستطيع إنشاء جيم، تعريف الخدمات والباقات، إضافة موظفين بصلاحيات، تسجيل العملاء، بيع الاشتراكات، تسجيل المدفوعات، تفعيل الاشتراكات، تسجيل الحضور، استخدام دعوات الضيوف، وبيع خدمات إضافية مستقلة.

3. الأدوار والصلاحيات

الأدوار الأساسية: SERVICE_PROVIDER صاحب جيم، EMPLOYEE موظف، CUSTOMER عميل.

الـ Backend لا يستقبل employeeId من الواجهة كممثل للعملية. هو يحدد المستخدم من JWT ثم يبحث هل هو صاحب الجيم أو موظف نشط له الصلاحية المطلوبة. صاحب الجيم له كل الصلاحيات. صلاحية ADMIN تكفي لكل عمليات الموظف.

الصلاحيات المتاحة: SELL_SUBSCRIPTION, ACTIVATE_SUBSCRIPTION, REGISTER_ATTENDANCE, REGISTER_INVITATION, SELL_EXTRA_SERVICE, MANAGE_SERVICES, MANAGE_PACKAGES, MANAGE_CUSTOMERS, MANAGE_EMPLOYEES, FREEZE_SUBSCRIPTION, MANUAL_ADJUSTMENT, CLOSE_CASH_DAY, ADMIN.

4. فلو التطبيق

  1. تسجيل مستخدم بدور SERVICE_PROVIDER.
  2. تسجيل الدخول وأخذ accessToken.
  3. إنشاء جيم، والمستخدم الحالي يصبح صاحب الجيم.
  4. تحديث إعدادات الجيم مثل الدفع الجزئي وسياسة التفعيل وطرق الدخول.
  5. إضافة موظفين وتحديد صلاحياتهم.
  6. إنشاء خدمات ثم إنشاء باقات تحتوي على بعض الخدمات.
  7. تسجيل العملاء أو ربط مستخدم موجود كعميل.
  8. بيع اشتراك للعميل وتسجيل الدفعة الأولى.
  9. لو الاشتراك مدفوع بالكامل وسياسة التفعيل IMMEDIATE يتم التفعيل تلقائيا. لو MANUAL يتم التفعيل من API التفعيل.
  10. تسجيل الحضور باستخدام كود العضوية أو رقم العميل.
  11. استخدام دعوات الضيوف من رصيد الاشتراك.
  12. بيع خدمات إضافية أو تسجيل استخدام خدمة مرفقة بالاشتراك.
